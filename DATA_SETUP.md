# Data Setup Guide

This document describes every external data artifact required to run SEMDT
and all extension experiments. No raw data is distributed with the repository.

---

## Required Data Artifacts

### 1. Processed nuPlan Scene Matrices

**Path:** `/data/nuscenes_fixed_matrices/`

These are pre-processed `.npy` or structured matrix files (one per scene) derived
from the nuPlan benchmark. Each file encodes a multi-agent scene as a fixed-capacity
array of shape `(T, N, F)` where:

- `T` — number of timesteps in the scene
- `N` — number of agents (padded to a fixed capacity, zeros for absent agents)
- `F = 10` — per-agent kinematic features: `[x, y, z, vx, vy, vz, heading, length, width, height]`

The ego vehicle is always at agent index `0`.

At **2 Hz sampling**, a `K=15` context window covers 7.5 seconds of history
and a `H=10` prediction horizon covers 5 seconds of future.

**How to obtain:**
These matrices are derived from the official [nuPlan dataset](https://www.nuscenes.org/nuplan).
You must register for data access at the nuPlan website, then process the raw logs
using the extraction scripts in `Archive/` (e.g., `extract_boston_singapore_scenes.py`).

The processed format expected by the training pipeline:
```
/data/nuscenes_fixed_matrices/
    scene_001.npy
    scene_002.npy
    ...
    scene_130.npy        # 130 scenes used in all paper experiments
```

### 2. Text Manifest (INTENT Captions + CLIP Embeddings)

**Path:** `/data/nuplan_text_finetuned/nuplan_text_manifest_vlm_metadata_enhanced.parquet`

A single Parquet file mapping each `(scene_id, frame_idx)` to:
- `text` — motion-aware INTENT caption (from BLIP-2 + GPT-2 refinement pipeline)
- `text_emb` — 512-dimensional CLIP text embedding (float32 array, L2-normalized)
- `scene_id` — scene identifier matching the filenames in the matrix directory
- `frame_idx` — timestep index within the scene

This file contains **6,500 text descriptions** covering all 130 scenes.

**How to generate (if you do not have the parquet file):**

```bash
# Step 1 — Generate motion-aware captions from nuPlan camera observations
python3 generate_prompted_metadata_captions.py

# Step 2 — Build combined manifest with CLIP embeddings baked in
python3 generate_gpt2_vlm_hybrid_manifest.py
```

Both scripts require:
- Access to raw nuPlan camera frames (available from the nuPlan dataset)
- A running BLIP-2 model (loaded via `transformers.Blip2Processor`)
- The GPT-2 refinement model (fine-tuned on driving descriptions — see below)

---

### 3. GPT-2 Caption Refinement Model (for caption generation only)

Used **offline** to refine BLIP-2 base captions with motion trend context.
This model is **not needed** at training time if the parquet manifest already exists.

The fine-tuned GPT-2 weights are generated by:

```bash
python3 finetune_gpt2_improved.py
```

This produces a GPT-2 model fine-tuned to produce driving-specific INTENT captions
when prompted with:
```
[BLIP-2 base caption] + [state trend: speed, acceleration, heading change]
→ "Write one sentence describing the ego vehicle's current maneuver and intent..."
```

## Data Split

All experiments use a fixed **80/20 scene-level** train/test split seeded with
`random.seed(42)` and `numpy.random.seed(42)`. Scene-level splitting ensures
no temporal leakage between train and test sets.

| Split | Scenes | Approximate samples |
|---|---:|---:|
| Train | 104 | ~3,100 trajectory windows |
| Test | 26 | ~780 trajectory windows |

Actual sample counts after movement/variance filtering vary per run — see
`RUNBOOK.md` for the filter values used in each experiment.

## State Vector Format

Each agent's 10 kinematic features are stacked across `N` agents per timestep
and flattened to produce the `state_dim` input to the model:

```
state_dim = N_agents × 10 features = 128 × 10 = 1,280 (paper setting)
                                   = 2,072 × 10 = 20,720 (server setting — larger scenes)
```

The model projects `state_dim → hidden_size` via a learned linear layer.

## Caption Granularity Manifests (Paper Q2 — Table I)

The paper evaluates three caption granularities to show that **semantic specificity
is safety-relevant** (Q2). Each granularity uses the same CLIP encoder and same
grounding mechanism — only the caption content changes:

| Granularity | What it encodes | Paper result |
|---|---|---|
| **OBJECT** | Object existence ("a car", "a pedestrian") | Reduces ADE but *increases* MR@5m |
| **SCENE** | Coarse scene context ("urban intersection") | Reduces both, less than INTENT |
| **INTENT** | Motion + interaction intent ("braking to yield") | Best: reduces both ADE and MR@5m |

To reproduce all three Table I rows, you need three separate parquet manifests.

The INTENT manifest is generated by `generate_prompted_metadata_captions.py`
(already included). To generate OBJECT and SCENE variants, modify the prompt
template in that script:

**OBJECT-level prompt:**
```python
prompt = f"Describe the objects visible in this driving scene: {blip2_caption}"
```

**SCENE-level prompt:**
```python
prompt = f"Briefly describe the driving scene context: {blip2_caption}"
```

**INTENT-level prompt (default — what the repo uses):**
```python
prompt = (
    f"{blip2_caption}. State trends: {state_trend}. "
    "Write one sentence describing the ego vehicle's current maneuver and intent, "
    "grounded in the motion trends and nearby traffic."
)
```

Save each variant to a separate parquet file and pass it via
`--text_manifest_path` when running the corresponding experiment.

## Notes

- The training code does **not** require raw video or camera data at runtime.
  All visual information is pre-baked into CLIP embeddings in the parquet manifest.
- If you are **only training** (not regenerating captions), you need only
  the two paths listed in section 1 and 2 above.
- Checkpoints are saved to `checkpoints/` in the working directory.
  They are excluded from git (see `.gitignore`).
